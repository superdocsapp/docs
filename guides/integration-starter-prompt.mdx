# Integration Starter Prompt

**Target fill time: under 5 minutes.** Paste your existing agent context (`CLAUDE.md`, `AGENTS.md`, `.cursorrules`), or a `package.json`, or nothing at all — the agent infers the rest and tells you what it picked before starting.

## What you'll build

A complete integration of SuperDocs into your product. Your coding agent reads this prompt, reads any context you paste into Part 0A (or infers from your codebase), confirms what it's inferred, and then writes the integration: editor chunk-id preservation, API proxy, streaming, inline diff overlay, Word export. The public SuperDocs docs are authoritative for every snippet the agent uses — so if you ever swap your editor or backend framework, the same prompt still works.

This template handles the realistic range of starting points:

- **Greenfield** with no app yet — the agent scaffolds one.
- **Existing app with no editor yet** — the agent adds one.
- **Existing app with an editor** — the agent wires SuperDocs into it.
- **Server-side / batch / agent-to-agent** — the agent skips the UI phases.

````text
## Agent protocol — inference, echo-back, confirm

Before executing **any** phase, the coding agent must:

1. **READ CONTEXT.** Read every file and URL provided in Part 0A. Then read the integrator's existing codebase — at minimum, the package manifest (`package.json`, `requirements.txt`, `Gemfile`, `go.mod`, `pyproject.toml`, etc.), the main framework config files, any editor component if declared or discoverable, and the `README` if present.
2. **INFER.** For every Part 0B slot the integrator left blank, infer the value from context. If a slot cannot be inferred with reasonable confidence, mark it as `UNSET`.
3. **ECHO BACK.** Present every Part 0B value with a source tag:
   - `[provided]` — the integrator explicitly set this.
   - `[inferred from <file or signal>]` — the agent inferred it from a specific source. Name the source.
   - `[UNSET — asking you]` — genuinely ambiguous, needs the integrator to decide.
4. **ASK.** For each `UNSET` slot, ask the integrator specifically, offering 2–3 suggested options.
5. **WAIT.** Do not execute Phase 1 until the integrator has confirmed the inferred values and answered the `UNSET` questions.

This prevents the agent from confidently integrating into the wrong stack because it misread pasted context or guessed at an ambiguous axis.

## Dry-run checklist (verify before first integration attempt)

Verify all five before you commit to a full integration pass. Any failure here is a 10× better thing to catch now than mid-build.

1. **Editor round-trip on real SuperDocs HTML.** Paste a realistic SuperDocs response (one with `data-chunk-id` on every block-level element) into the editor, read it back, confirm the `data-chunk-id` attribute survives the round-trip on every block node type your editor supports. 2. **Scaffold / install commands are current for your declared stack.** The snippets in this template reflect versions at publication time; languages, frameworks, and SDKs drift. If a command or flag is rejected, drop the offending flag, continue. Do not stop to debug the scaffolder.
3. **Styling plugin compatibility.** If you chose Tailwind, Next.js 16+ ships Tailwind v4 by default. If you're on Tailwind v3, the plugin import syntax differs — see Phase 2 note. If you're not using Tailwind at all, ignore `.prose` class references and style with your own system.

If one specific line throws a type error, add a targeted cast at the usage site — don't refactor the helper.
5. **SSE consumption works in your declared frontend.** Send one test message, confirm at least one `intermediate` event arrives before wiring the rest of the UI. A silent SSE connection is the #1 time-burn.

## Part 2 — Hard constraints

### Scope — what the integration must produce
- Users can send a natural-language instruction, see proposed AI edits, review them, and have approved edits applied to their document in place.
- The AI's proposed changes are visible to the user before they apply (HITL review mode).
- The integration does not expose your SuperDocs `sk_` API key to the browser.
- The document HTML (with `data-chunk-id` attributes) round-trips correctly between your editor and SuperDocs.

### Scope — what is explicitly out of scope for a first integration
- Attachments, image uploads, and templates. All exist in the API; they're separate integrations.
- MCP. That's a different transport — set `DEPLOYMENT_TARGET=mcp-only` and follow the MCP guide instead.
- Multi-document UI, session list, user impersonation. Not part of the API integration.

### Content safety — what to never include in your code, comments, or public-facing artefacts
- Specific AI model names (the SuperDocs API includes the model; your users do not need to know which one).
- Third-party orchestration framework names that SuperDocs might use internally.
- Cloud provider names beyond "hosted" or "server-side."
- Database technology assumptions.
- Cost-per-operation numbers or internal SuperDocs pricing.

Say "SuperDocs' AI" or "the AI" when you need to refer to the model.

### Phase 2 — Editor setup with SuperDocs chunk-id preservation

**Run only if:** `EDITOR_STATE = none` OR (`EDITOR_STATE ∈ {installed-basic, installed-custom}` AND chunk-id preservation not yet wired).

**Background.** SuperDocs tags every block-level element in returned HTML with `data-chunk-id="<uuid>"`. Every serious rich-text editor strips unknown attributes by default unless its schema or config tells it otherwise. The failure mode is silent: surgical editing works intermittently and fails mysteriously. This phase wires up the preservation on your chosen editor.

Pick the snippet matching `EDITOR_FRAMEWORK`:

#### ProseMirror (raw)

Install: `npm install prosemirror-state prosemirror-view prosemirror-model prosemirror-schema-basic prosemirror-schema-list prosemirror-history prosemirror-keymap prosemirror-commands`.

```typescript
// lib/schema.ts
import { Schema, NodeSpec } from "prosemirror-model";
import { schema as basicSchema } from "prosemirror-schema-basic";
import { addListNodes } from "prosemirror-schema-list";

function withChunkId(spec: NodeSpec): NodeSpec {
  const attrs = { ...(spec.attrs ?? {}), "data-chunk-id": { default: null } };
  const parseDOM = (spec.parseDOM ?? []).map((rule) => ({
    ...rule,
    // If your TS complains about this signature, cast with `as HTMLElement`
    // at the usage site — do NOT refactor the signature.
    getAttrs: (node: string | HTMLElement) => {
      const base = typeof rule.getAttrs === "function"
        ? rule.getAttrs(node as HTMLElement) ?? {}
        : rule.attrs ?? {};
      if (typeof node === "string") return base;
      return { ...base, "data-chunk-id": node.getAttribute("data-chunk-id") };
    },
  }));
  const originalToDOM = spec.toDOM;
  const toDOM: NodeSpec["toDOM"] = originalToDOM
    ? (node) => {
        const out = originalToDOM(node);
        if (!Array.isArray(out)) return out;
        const [tag, maybeAttrs, ...rest] = out as [string, unknown, ...unknown[]];
        const chunkId = node.attrs["data-chunk-id"];
        if (!chunkId) return out;
        const isAttrs = maybeAttrs && typeof maybeAttrs === "object"
          && !Array.isArray(maybeAttrs)
          && !(maybeAttrs as { nodeType?: unknown }).nodeType;
        return isAttrs
          ? [tag, { ...(maybeAttrs as Record<string, unknown>), "data-chunk-id": chunkId }, ...rest]
          : [tag, { "data-chunk-id": chunkId }, maybeAttrs, ...rest];
      }
    : undefined;
  return { ...spec, attrs, parseDOM, toDOM };
}

let nodes = basicSchema.spec.nodes;
for (const name of ["paragraph", "blockquote", "heading", "horizontal_rule", "code_block"]) {
  const spec = nodes.get(name);
  if (spec) nodes = nodes.update(name, withChunkId(spec));
}
nodes = addListNodes(nodes, "paragraph block*", "block");
for (const name of ["ordered_list", "bullet_list", "list_item"]) {
  const spec = nodes.get(name);
  if (spec) nodes = nodes.update(name, withChunkId(spec));
}

export const schema = new Schema({ nodes, marks: basicSchema.spec.marks });
```

#### Slate

Install: `npm install slate slate-react slate-history slate-hyperscript`.

```typescript
// lib/slate-chunk-id.ts
import { Element as SlateElement } from "slate";

export type ChunkedElement = SlateElement & { "data-chunk-id"?: string | null };

// Serialize a Slate document to HTML, preserving data-chunk-id
export function toHtml(nodes: ChunkedElement[]): string {
  return nodes.map((n) => {
    const id = n["data-chunk-id"] ? ` data-chunk-id="${n["data-chunk-id"]}"` : "";
    const tag = n.type ?? "p";
    const inner = (n.children as { text?: string }[])
      .map((c) => c.text ?? "").join("");
    return `<${tag}${id}>${inner}</${tag}>`;
  }).join("");
}

// Parse SuperDocs HTML back into Slate nodes with the chunk id preserved
export function fromHtml(html: string): ChunkedElement[] {
  const doc = new DOMParser().parseFromString(html, "text/html");
  return Array.from(doc.body.children).map((el) => ({
    type: el.tagName.toLowerCase(),
    "data-chunk-id": el.getAttribute("data-chunk-id"),
    children: [{ text: el.textContent ?? "" }],
  })) as ChunkedElement[];
}
```

Render with your existing `renderElement` — just pass `data-chunk-id` through on the outer element.

#### Lexical

Install: `npm install lexical @lexical/react @lexical/html`.

```typescript
// lib/lexical-chunk-id.ts
import { $getRoot, $createParagraphNode, LexicalNode } from "lexical";
import { $generateHtmlFromNodes, $generateNodesFromDOM } from "@lexical/html";

// Monkey-patch exportDOM on each block node type you care about to keep data-chunk-id.
// The cleanest pattern is a transform + element attribute map kept on a WeakMap.
const chunkIdMap = new WeakMap<LexicalNode, string>();

export function setChunkId(node: LexicalNode, id: string | null) {
  if (id) chunkIdMap.set(node, id); else chunkIdMap.delete(node);
}
export function getChunkId(node: LexicalNode) {
  return chunkIdMap.get(node) ?? null;
}

// When importing SuperDocs HTML:
export function importHtml(editor, html: string) {
  editor.update(() => {
    const doc = new DOMParser().parseFromString(html, "text/html");
    const nodes = $generateNodesFromDOM(editor, doc);
    // Walk DOM + nodes in parallel; when a DOM element has data-chunk-id, record it on the node.
    doc.body.querySelectorAll("[data-chunk-id]").forEach((el, i) => {
      const id = el.getAttribute("data-chunk-id");
      if (nodes[i] && id) setChunkId(nodes[i], id);
    });
    $getRoot().clear().append(...nodes);
  });
}

// When exporting to HTML for SuperDocs:
export function exportHtml(editor) {
  let html = "";
  editor.getEditorState().read(() => {
    html = $generateHtmlFromNodes(editor, null);
    // Post-process: re-attach data-chunk-id based on chunkIdMap. Walk block nodes
    // and inject the attribute into the generated HTML.
  });
  return html;
}
```

Lexical requires the most custom wiring because `data-chunk-id` isn't a native concept in its model. Verify round-trip carefully during the dry-run.

#### ContentEditable (no editor library)

```typescript
// lib/contenteditable-chunk-id.ts
// A plain contenteditable div does preserve attributes by default, but some
// browsers silently re-normalize HTML when the user types. Protect against
// that with a MutationObserver that restores missing chunk ids from a
// snapshot taken before each AI round-trip.

const lastSeenIds = new Map<Element, string>();

export function snapshotChunkIds(root: Element) {
  root.querySelectorAll("[data-chunk-id]").forEach((el) => {
    const id = el.getAttribute("data-chunk-id");
    if (id) lastSeenIds.set(el, id);
  });
}

export function restoreChunkIds() {
  for (const [el, id] of lastSeenIds.entries()) {
    if (el.isConnected && !el.getAttribute("data-chunk-id")) {
      el.setAttribute("data-chunk-id", id);
    }
  }
}

// Install a MutationObserver on the editable root and call restoreChunkIds()
// whenever a childList or attributes mutation fires.
```

### Phase 4 — API-key-safe proxy to SuperDocs

**Always runs.**

**Background.** The SuperDocs `sk_` key is a secret. Never expose it to a browser. Your backend holds it in an env var and forwards calls to `https://api.superdocs.app`, injecting the `Authorization: Bearer sk_...` header (or `api_key=sk_...` query param for the SSE stream, because `EventSource` can't set headers).

Pick the snippet matching `BACKEND_STATE`:

#### Next.js App Router (TypeScript)

```typescript
// app/api/superdocs/[...path]/route.ts
import { NextRequest } from "next/server";

const BASE = "https://api.superdocs.app";
const KEY = process.env.SUPERDOCS_API_KEY;

async function proxy(req: NextRequest, ctx: { params: Promise<{ path: string[] }> }) {
  if (!KEY) return new Response("Missing SUPERDOCS_API_KEY", { status: 500 });
  const { path } = await ctx.params;
  const target = `${BASE}/${path.join("/")}`;
  const headers = new Headers(req.headers);
  headers.set("Authorization", `Bearer ${KEY}`);
  headers.delete("host");
  const body = req.method === "GET" || req.method === "HEAD"
    ? undefined : await req.arrayBuffer();
  const res = await fetch(target, { method: req.method, headers, body });
  return new Response(res.body, { status: res.status, headers: res.headers });
}

export const GET = proxy;
export const POST = proxy;
export const DELETE = proxy;
```

```typescript
// app/api/superdocs/stream/route.ts
import { NextRequest } from "next/server";
const BASE = "https://api.superdocs.app";
const KEY = process.env.SUPERDOCS_API_KEY!;

export async function GET(req: NextRequest) {
  const url = new URL(req.url);
  const jobId = url.searchParams.get("job_id");
  const sid = url.searchParams.get("sid");
  if (!jobId || !sid) return new Response("Missing job_id or sid", { status: 400 });
  const target = `${BASE}/v1/chat/${encodeURIComponent(sid)}/stream` +

package main

import ( "io"; "net/http"; "os"; "strings" )

var key = os.Getenv("SUPERDOCS_API_KEY")
const base = "https://api.superdocs.app"

func proxy(w http.ResponseWriter, r *http.Request) {
    target := base + strings.TrimPrefix(r.URL.Path, "/api/superdocs")
    req, _ := http.NewRequest(r.Method, target, r.Body)
    req.Header = r.Header.Clone()
    req.Header.Set("Authorization", "Bearer "+key)
    resp, err := http.DefaultClient.Do(req)
    if err != nil { http.Error(w, err.Error(), 502); return }
    defer resp.Body.Close()
    for k, v := range resp.Header { w.Header()[k] = v }
    w.WriteHeader(resp.StatusCode)
    io.Copy(w, resp.Body)
}

func stream(w http.ResponseWriter, r *http.Request) {
    q := r.URL.Query()
    url := base + "/v1/chat/" + q.Get("sid") + "/stream?job_id=" + q.Get("job_id") + "&api_key=" + key
    req, _ := http.NewRequest("GET", url, nil)
    req.Header.Set("Accept", "text/event-stream")
    resp, err := http.DefaultClient.Do(req)
    if err != nil { http.Error(w, err.Error(), 502); return }
    defer resp.Body.Close()
    w.Header().Set("Content-Type", "text/event-stream")
    w.Header().Set("Cache-Control", "no-cache, no-transform")
    w.WriteHeader(resp.StatusCode)
    flusher, _ := w.(http.Flusher)
    buf := make([]byte, 4096)
    for {
        n, err := resp.Body.Read(buf)
        if n > 0 { w.Write(buf[:n]); if flusher != nil { flusher.Flush() } }
        if err != nil { return }
    }
}

### Phase 6 — `document_sync` handling

**Always runs.**

When you send `document_html`, SuperDocs emits a `document_sync` event at the start of the stream containing the prepared HTML with chunk IDs. Apply it to your editor immediately so the editor's chunk IDs match what the AI references in `proposed_change` events that follow.

This is two lines: in your stream handler, wire `document_sync` → `editor.setHtml(content)`. Already included in the Phase-5 snippets above.

**Tradeoff if you skip it:** edits still apply via `final`, but mid-flight diff cards (Phase 7) may reference chunk IDs your editor doesn't have yet. Symptom: diff cards render but "which section is this?" anchoring breaks for freshly pasted/uploaded documents. Worth the two lines.

### Phase 7 — HITL review drawer

**Always runs.**

**UI style is yours.** Do not pull in Tailwind / shadcn / MUI just for this. Use whatever component library you already have. The logic below is what matters; skin it to your app.

On chat send: include `approval_mode: "ask_every_time"` in the `POST /v1/chat/async` body. SuperDocs will emit one `proposed_change` event per AI-proposed edit, then pause.

On each `proposed_change` (already wired in Phase 5):
1. Parse `content` as JSON — fields are `change_id`, `operation` (`edit`/`create`/`delete`), `chunk_id`, `old_html`, `new_html`, `ai_explanation`, `batch_id`, `batch_total`, `insert_after_chunk_id`.
2. Push it onto a `pending` list in your chat state.
3. Render a review card per change: show `ai_explanation`, and `old_html` / `new_html` side-by-side with red and green backgrounds. Three buttons: Approve, Deny, (optional) Deny with feedback.

On approve/deny:

```javascript
await fetch(`/api/superdocs/v1/chat/${sessionId}/approve`, {
  method: "POST",
  headers: { "Content-Type": "application/json" },
  body: JSON.stringify({
    job_id: currentJobId,

After the user decides, the AI resumes on the already-open SSE stream — the `approve` POST itself just returns 200.

On `final`: the `result.document_changes.updated_html` replaces the editor's content. Clear the pending list.

## Part 5 — API contract summary

All requests go through your proxy. Paths are relative to `/api/superdocs/`.

- `POST v1/chat/async` — body: `{ message, session_id, document_html, approval_mode?: "approve_all" | "ask_every_time" }` → `{ job_id }`.
- `GET stream?job_id=...&sid=...` — SSE stream of six event types.
- `POST v1/chat/{session_id}/approve` — body: `{ job_id, approved, changes?: [{ change_id, approved, feedback? }] }`.
- `POST v1/documents/export` — body: `{ html?, session_id?, format: "doc"|"docx", filename? }` → binary Word file.

SSE event `data:` shapes (all JSON):

```
document_sync   { type, content }                  // content: full HTML
intermediate    { type, content, sequence, timestamp }
proposed_change { type, content, sequence }        // content: JSON-stringified ProposedChange
final           { content, result }                // result.response, result.document_changes.updated_html
usage           { monthly_used, monthly_limit, monthly_remaining, was_billable, subscription_tier }
error           { error }
```

Parsed `proposed_change.content` shape:
```
{ change_id, operation: "edit" | "create" | "delete",
  chunk_id, old_html, new_html,
  ai_explanation, batch_id, batch_total, insert_after_chunk_id }
```

## Part 6 — Failure modes and their fixes

| Symptom | Likely cause | Fix |
|---|---|---|
| `Missing SUPERDOCS_API_KEY` | env file not read | Restart your server; env vars are read on boot. |
| 401 from `/v1/chat/async` | key wrong or includes `Bearer ` literal | Value is just `sk_...` with no quotes, no "Bearer" prefix. |
| SSE connection opens then closes | proxy transforms headers | Pipe upstream body unchanged; set `Cache-Control: no-cache, no-transform`. |
| `proposed_change` fires, approve 404 | wrong session_id in URL | Path must be `/api/superdocs/v1/chat/${sessionId}/approve`. |
| Editor loses `data-chunk-id` after first edit | editor schema/config missing a node type | Add the missing node to your chunk-id-preservation config (Phase 2). |
| Diff card `old_html` empty | operation is `create`, not `edit` | Correct — creates have no `old_html`. Render them as "new section" cards. |
| Silent round-trip loss | HTML sanitizer strips unknown attributes | Add `data-chunk-id` to your sanitizer's allowlist. |
| SSE works in dev, fails in prod | reverse proxy buffers | Disable buffering for `text/event-stream` in nginx/Caddy/CloudFront. |
````
